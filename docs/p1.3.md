## Task 3 – User Index: BST by Follower Count

### 1. Why is a BST useful here alongside the graph? What does it give you that the graph alone cannot?

While the directed graph data structure excels at capturing structural dependencies and relational mappings (such as tracking follower/following vectors, determining degree distributions, or executing pathfinding traversals like BFS), it is inherently inefficient for non-relational, attribute-based querying. 

If the system relies solely on the social graph to retrieve ranked metrics or execute bounded filters, it would require a full graph traversal ($O(N)$ linear scan), where $N$ represents the total number of user nodes. In a production-level network with millions of accounts, this architectural bottleneck would severely degrade performance.

By maintaining a Binary Search Tree (BST) indexed by `follower_count` alongside the graph, we instantiate an auxiliary ranked database index. The BST provides capabilities that the graph structure alone cannot support efficiently:
- **Efficient Ranked Lookups:** Instead of iterating through all nodes, searching for an exact follower metric can be performed in $O(\log N)$ average time complexity.
- **Logarithmic Range Filtering:** Finding all user objects within a specific demographic window $[min, max]$ can be evaluated in $O(\log N + M)$ time (where $M$ is the number of matching elements) by pruning non-matching subtrees.
- **Instant Top-K Extraction:** Retrieving the top-K most influential content creators can be computed deterministically via a reverse in-order traversal, bypassing the need to sort the entire database.

### 2. What happens to your BST if follower counts change dynamically?

In a standard Binary Search Tree, the strict spatial invariant dictates that for any given node, all keys within its left subtree must be strictly less than the node's key, and all keys within its right subtree must be strictly greater. 

If a user's `follower_count` increments or decrements dynamically (e.g., a viral post causes a surge in followers), the user's data record breaches the local BST invariant. If left unadjusted, the node becomes misplaced, corrupting the search property and breaking subsequent retrieval pipelines.

To handle dynamic real-time count mutations robustly, the engineering pipeline operates as follows:
1. **Extraction (De-indexing):** Before updating the user's account state, the system searches for the user's old `follower_count` within the BST and removes that specific `User` object from the node's list. If the node's user bucket becomes empty, standard BST node deletion is triggered.
2. **State Mutation:** The user’s core attribute value is updated with the new follower count.
3. **Re-insertion (Re-indexing):** The system invokes `BST.insert()` using the updated count, traversing down the tree to position the user object inside the correct, newly designated tree boundary.

This maintains structural alignment with $O(\log N)$ computational overhead per update event.
